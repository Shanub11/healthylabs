HealthyLabs Medical Data Extractor 
FINAL COMPREHENSIVE EXTRACTION STRATEGY (WITH ALL FIXES) 
PART 0: EXECUTIVE SUMMARY 
This document defines the medical-grade data extraction strategy for HealthyLabs.ai. The extractor is a 
neutral preservation layer that feeds the downstream pipeline with high-fidelity content, rich structure, 
and clear quality signals—without interpretation. 
Core Principles: 
• Preserve everything - Never discard source information 
• Remain neutral - No interpretation, no trust-tier judgment 
• Surface structure - Layout, hierarchy, relationships 
• Signal quality - Confidence scores, warnings, audit trails 
• Enable multi-modal - Images preserved and linked to text 
Output Guarantees: 
• Lossless image preservation (original formats retained) 
• Complete audit trail (chain of custody) 
• Rich structural annotations (layout graph, hierarchy, citations) 
• Quality transparency (OCR confidence, extraction warnings) 
• Bidirectional linking (text ↔ images, layout ↔ hierarchy) 
PART 1: SYSTEM ARCHITECTURE 
1.1 Core Design Philosophy 
The extractor is a medical-grade intake valve with a single responsibility: preserve every byte of 
information with absolute fidelity while providing clear structural signals to downstream systems. It 
operates independently of the refinery pipeline and remains trust-tier agnostic—judgment happens later.

1.2 The Four Stages 
Stage 1: Fetcher (Source-Agnostic Downloader) 
Responsibility: Authenticate and download files from diverse sources with retry logic. 
Supported Sources: 
• Google Drive (via API with OAuth) 
• Public HTTPS/HTTP URLs 
• Amazon S3 / S3-compatible storage (MinIO) 
• SFTP servers 
• Local file system mounts 
• Email attachments (IMAP) 
Stage 2: Validator (Deduplication & Integrity) 
Responsibility: Prevent redundant processing and verify file integrity. 
Core Functions: 
• SHA-256 hash computation of binary file before any preprocessing 
• Check against metadata store (PostgreSQL) 
• Skip if hash exists with complete audit trail 
• Log duplicate attempts for tracking 
Document ID Definition: 
document_id = sha256-of-original-binary-file 
• Hash computed on raw bytes as received 
• Stable across re-ingestion attempts 
• Primary key for all document references 
Stage 3: Polymorphic Extraction Engine 
Responsibility: Convert any binary input into a standardized, high-fidelity output package. 
Output Package Structure: 
{ 
"document_id": "a1b2c3...", 
"content": "markdown_text_with_placeholders", 
"images": [...], 
"tables": [...], 
"layout_graph": {...}, 
"section_hierarchy": {...}, 
"citations": {...}, 
"medical_patterns": {...}, 
"metadata": {...}, 
"quality": {...},

Stage 4: Delivery Agent 
Responsibility: Package and deliver to pipeline with retry logic. 
Features: 
• Schema validation before sending 
• Exponential backoff retry (3 attempts) 
• Dead letter queue for permanent failures 
• Delivery confirmation tracking

4.2 Text-Containing Images: Never Discard 
Rule: If an image contains text, preserve the original AND provide OCR. 
Why: 
• OCR errors happen (μ vs μ, 10mg vs 100mg) 
• Layout conveys meaning (superscripts, subscripts) 
• Dosage formatting may be visual 
• Audit trail requires original 
"text_snippet_image": { 
"image_id": "a1b2c3_img_7_3", 
"original_path": "s3://images/a1b2c3_img_7_3.jpg", 
"ocr_text": "Take 10 mg twice daily", 
"ocr_confidence": 0.94, 
"ocr_engine": "tesseract-5.3.0", 
"layout_notes": "superscript detected on 'mg'" 
} 
4.3 Image Enhancement: Geometric Only 
Allowed (Safe): 
• Deskew (affine transform only) 
• Despeckle (morphological filtering) 
• Contrast stretching (histogram adjustment) 
• Binarization (thresholding algorithms) 
• Sharpening (unsharp mask - kernel-based) 
Prohibited (Unsafe): 
• Neural upscaling (can hallucinate strokes) 
• Generative inpainting (creates fake data) 
• Super-resolution ML (adds false details) 
• Any ML that modifies text shapes 
PART 5: OCR & TEXT EXTRACTION 
5.1 OCR Pipeline 
Scanned PDF Processing 
1. Preprocessing (geometric only): 
a. Deskew 
b. Despeckle 
c. Contrast enhancement 
d. Resolution check (target 300-600 DPI) 
2. OCR Engine: Tesseract 5 with medical language pack 
a. Custom-trained on medical terminology 
b. Pharmaceutical term dictionary 
c. Greek/Math symbol support 
3. Confidence Scoring: 
a. Word-level confidence from Tesseract 
b. Page-level aggregation 
c. Document-level summary

5.3 Digital Text Extraction 
Primary Tool: pypdfium2 (Chromium PDF engine) 
Fallback Chain: 
1. pypdfium2 
2. pypdf (if pypdfium2 fails) 
3. pdfminer.six (text-only fallback) 
5.4 Character Encoding 
Problem: Medical texts contain Greek letters (β, μ), math symbols (≤, ≥), diacritics (é, ñ), and special 
characters (§, †). 
Solution: 
1. Detect encoding with charset-normalizer 
2. Convert to UTF-8 
3. Normalize Unicode form (NFC) 
4. Validate against medical symbol dictionary 
5. Log any replacements or failures

PART 11: EDGE CASE HANDLING 
11.1 Document Size Extremes 
Scenario Risk Strategy 
1,000+ page textbook Memory exhaustion Page-streaming, process sequentially 
1GB+ PDF Timeout Chunked download, streaming parse 
10,000+ images Storage spike Immediate S3 upload, don't hold in memory 
11.2 Corruption Scenarios 
Scenario Strategy 
Corrupted PDF Fallback chain (pypdfium2 → pypdf → pdfminer) 
Truncated file Detect size mismatch, quarantine 
Malformed XML defusedxml safety, quarantine on error 
Infinite loop Hard timeout per document (10 min max) 
11.3 Security Scenarios 
Scenario Strategy 
Password protected Quarantine, never attempt guessing 
Suspicious content Sandboxed extraction, resource limits 
Zip bombs Detect decompression ratio, abort 
Script in PDF Isolated environment, disable JavaScript 
11.4 Medical Edge Cases 
Scenario Strategy 
Retracted publication Check against retraction DB, flag in metadata 
Supplementary materials Group with main document 
Non-English text Language detection, preserve original 
Handwritten annotations Detect, preserve image, flag for human review 
Watermarks Detect, preserve, log presence
